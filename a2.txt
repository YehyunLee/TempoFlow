Dancer alignment validation Data Generation Pipeline

The Dancer alignment pipeline creates validation datasets by applying mathematical affine transformations (such as scaling, rotation, and aspect ratio changes) to reference dance videos. This ensures the algorithm evaluates the user's kinematic performance rather than their recording environment, guaranteeing that scores are unaffected by camera placement, distance from the lens, or video resolution differences.

Data Processing Steps
Step 1: Core Motion Selection Source:
Source: A random 3–5 second clip (T) is sampled from a high-quality dance dataset ( AIST++).
Constraint: The clip must contain high-variance motion (i.e., not standing still) to ensure the algorithm has active movement to score.
Step 2: Parallel Transformation:
The reference clip is duplicated to create a "Student" clip (Clip S), which is processed via 2D affine transformations to simulate specific recording variances:
Spatial Simulation: We apply randomized matrix operations including Scaling (0.8x–1.2x to simulate distance), Translation (±10% to simulate off-center positioning), and Rotation (±15∘ to simulate crooked camera angles).
Morphological Simulation: We alter the aspect ratio (width stretch 0.9x–1.1x) to act as a proxy for validating robustness against different body builds and camera driver distortions.
Constraint: All transformations must ensure the dancer remains fully within the frame boundaries to prevent MediaPipe body detection failures.
Step 3: Temporal & Distractor Injection:
To validate temporal sensitivity and negative matching, we generate specific "failure" cases:
Drift Injection: We trim the start of the Student clip by +0.1s to +0.5s. The algorithm must detect this offset and lower the score accordingly.
Negative Sampling: A completely distinct clip is selected from the dataset (Clip D). The algorithm must score the pair (T vs. D) near 0%.
Step 4: Ground Truth Export
For every generated pair, a JSON artifact is created containing the S3 keys for the Reference and Student clips, the transformation type applied, and the expected score range.

Each transformed video's ID prepends its parent reference ID followed by a transform index (R001_T001, R001_T002, …) followed by its transformation type, making the relationship between videos and the transformation made immediately clear.

Decisions and Trade Offs
Decision 1: Synthetic 2D Transformations vs. Real "Bad" Dancers
Decision: We utilize mathematical 2D affine transformations (rotation matrices, scaling) on the reference video rather than recording real humans performing errors.
Rationale: Precision. We can mathematically guarantee that Clip B is exactly "Clip A rotated 15 degrees." This isolates the variable being tested. If we used real videos, a low score could be due to timing, limb length, or balance, making root-cause analysis impossible.
Trade-off: Synthetic transformations lack biological physics. A "stretched" video does not perfectly mimic a different body type, and a "time-shifted" video does not capture the loss of balance a human experiences when missing a beat.

Decision 2: Aspect Ratio Distortion vs. 3D Body Simulation
Decision: We simulate different body types (heights/weights) by altering the video aspect ratio (stretching/squashing) rather than using complex 3D mesh simulations.
Rationale: This serves as a sufficient proxy to verify that the pose normalization layer (which should mathematically standardize height and width) is functioning correctly. It avoids the complexity of generating synthetic 3D humans.
Trade-off: Visual realism. The video will look unnaturally distorted to a human observer, even though the skeleton data remains valid for the algorithm.

Decision 3: Multiple Transformation Stress Test
Decision: We include a subset of test cases where all transformations (Scale + Rotate + Translate + Aspect Ratio) are applied simultaneously.
Rationale: Unit tests catch isolated issues from one type of transformation, but Combination transformation tests ensure the normalization pipeline doesn't suffer from compounding errors (e.g., a rotation making a translation calculation incorrect).
Trade-off: Debugging complexity. If a "Combo" test fails, it is difficult to determine which specific transformation caused the failure without reverting to the isolated test cases.

Decision 4: Streaming Ingestion (Presigned URLs) vs. Bulk Download
Decision: The validation worker streams video data directly from S3 using Presigned URLs, rather than downloading the dataset to disk.
Rationale: Latency and startup time. Downloading the full dataset (gigabytes of videos) would make testing slow to start. Streaming allows the scoring engine to process frames immediately as they arrive.
Trade-off: Network Jitter. If the S3 connection lags, the video decoder might drop frames or stall. We must implement "Blocking Read" logic in the loader to ensure the scoring algorithm doesn't penalize the dance for what is actually a network issue.

Audio Alignment Algorithm Validation Data Generation Pipeline.

The Audio Alignment pipeline generates validation datasets by degrading audio clips with environmental noise and frequency filters while maintaining precise timestamp records. Its purpose is to verify that the algorithm can accurately synchronize a user's recording to a reference track based solely on audio content, ensuring robustness against variations in microphone quality or background interference.

Data Processing Steps

Step 1: Core Signal Selection
Source: A random track (T) is sampled from the GTZAN dataset.
Constraint: The entire clip is treated as the "Match Window" to maximize feature density.

Step 2: Parallel Degradation
The audio track is duplicated to make two clips (Clip A and Clip B), which are processed independently to ensure spectral divergence:
Acoustic Simulation (EQ): Each branch applies a randomized Bandpass Filter (e.g., Clip A: 200Hz–3kHz; Clip B: 100Hz–5kHz) to simulate different microphone frequency responses.
Environmental Injection: randomly selected audio clips from DEMAND are mixed into both clips at randomized Signal-to-Noise Ratios (SNR) ranging from 5dB to 20dB.

Step 3: Temporal Shifting
To generate the target offsets, we append "Distractor" audio to the start and end of both clips.
Method: Random segments are sampled from the source dataset (distinct from T) and concatenated to the start/end of the degraded clips.
Result: The algorithm must distinguish the target audio from semantically similar "distractor" audio, rather than simply detecting the onset of noise vs. silence.

Step 4: Ground Truth Export
For every generated pair, a JSON artifact is created containing the precise sample indices where T begins and ends in both files, providing a deterministic answer key for testing.

Decisions and Trade Offs

Decision 1: Semantic "Distractor" Padding vs. Silence/Noise
Decision: We utilize random music/speech clips from the dataset as padding rather than silence or Gaussian noise.
Rationale: This prevents the algorithm from cheating by detecting simple energy onsets (i.e., finding where silence stops). It forces the model to perform true content matching, distinguishing the target signal from other valid audio signals.
Trade-off: If the algorithm fails, it is harder to diagnose why. With silence padding, failure is obvious (it didn't find the signal). With semantic padding, a failure could mean the algorithm correctly found a "similar" spectral feature in the distractor audio (a False Positive), making root-cause analysis significantly more time-consuming.

Decision 2: Environmental Noise vs. Synthetic (White) Noise
Decision: We inject real-world environmental recordings (non-stationary noise) rather than mathematically generated White/Pink noise.
Rationale: Synthetic noise is stationary (constant mean/variance) and easily removed by standard filters. Real-world noise (e.g., a car horn, a distinct voice) has dynamic time-frequency features that overlap with the target, providing the only valid test for real-world robustness.
Trade-off: Unlike Gaussian noise which is generated via one line of code (np.random.normal), this approach requires curating, cleaning, and normalizing a secondary "Noise Dataset," adding dependency management and build complexity to the pipeline.

Decision 3: Double-Blind Degradation
Decision: Both files in the pair are degraded; neither is a "clean" reference.
Rationale: This mirrors peer-to-peer alignment scenarios (e.g., syncing two user videos) where no "master" copy exists. It validates that the algorithm will work for the actual audio clips that are uploaded.
Trade-off: When both signals are heavily degraded, the "true" mathematical peak of cross-correlation becomes wider and flatter. This inherently limits the maximum theoretical time-resolution we can validate (e.g., we may only be able to certify alignment to within ±50ms rather than ±1ms), as the "exact" match point becomes acoustically ambiguous.

Decision 4: Cloud Storage (S3) vs. Local Storage
Decision: We will host the master datasets and generation pipeline on AWS S3, transitioning away from pure local storage.
Rationale: As the team and project scale, sharing 1.7GB+ datasets via Google Drive or git-LFS becomes unmanageable. S3 provides a centralized, versioned, and programmatic way to manage data. It allows us to trigger validation jobs automatically in the cloud (e.g., via AWS Batch or Lambda) without requiring an engineer's laptop to be running.
Trade-off: This introduces network latency and AWS costs (storage + egress). However, the cost is negligible for this data size (<5GB), and the consistency benefits outweigh the latency.
Decision 5: Hybrid Development Workflow (Local Caching)
Decision: While S3 is the master, we will maintain tooling to run the pipeline entirely locally or cache S3 data locally.
Rationale: Requiring a cloud upload/download for every single code change slows down the "write code -> test" loop. By allowing the generator to write to a local ./output directory during development, we reduce the amount of time we will have to spend waiting when testing our models for run enabling faster iteration.
Trade-off: This requires maintaining two "modes" in our scripts (e.g., a --local flag vs. a --s3 flag), which adds slight code complexity to the file I/O handlers.
